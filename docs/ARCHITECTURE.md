# Архітектура Lar’s Cloud

## Технології

| Частина | Рішення | Причина |
|---|---|---|
| Windows UI | WPF, .NET 8 | Нативне Windows-вікно, анімації, низьке навантаження в очікуванні |
| Google OAuth | REST OAuth 2.0, Authorization Code + PKCE | Системний браузер, пароль не потрапляє до програми |
| Google Drive | Drive API v3 через `HttpClient` | Прямий канал комп’ютер → Google, resumable upload |
| Локальний стан | SQLite (`Microsoft.Data.Sqlite`) | Атомарний облік файлів та історії |
| Захист токена | Windows DPAPI CurrentUser | Токен непридатний для іншого Windows-користувача/комп’ютера |
| Фонова робота | асинхронний scheduler + системний трей | Немає постійного сканування та навантаження CPU |
| Інсталятор | Inno Setup 6 | Один Setup.exe, ярлики, автозапуск, uninstall |
| Оновлення | GitHub Releases + SHA-256 | Версовані готові Installer-файли, перевірка цілісності |

## Потік синхронізації

1. Перевірити локальну папку, інтернет і Google-сесію.
2. Просканувати папку один раз; reparse points пропускаються, щоб уникнути циклів.
3. Звірити шлях, розмір і UTC-час зміни з SQLite.
4. Для нового/зміненого файлу обчислити SHA-256.
5. Перевірити доступне місце Drive.
6. Створити або знайти `Lar's Cloud / НАЗВА_КОМП'ЮТЕРА / НАЗВА_ЛОКАЛЬНОЇ_ПАПКИ` та її вкладені папки.
7. Для кожного кандидата знайти наявний Drive ID або файл із тим самим ім’ям у правильній батьківській папці.
8. Створити resumable upload session і передавати файл частинами по 8 МБ.
9. Після підтвердження Google атомарно оновити SQLite-запис.
10. Записати історію, оновити дати та показати сповіщення.

Якщо процес перервано до кроку 9, файл під час наступної синхронізації буде безпечно звірено повторно. Незмінені файли не передаються.

## Видалення

Типово видалення локального файлу не виконує жодної операції на Drive. Якщо користувач явно вмикає відповідний перемикач, Lar’s Cloud бере Drive ID лише з власної SQLite-бази, видаляє конкретний відомий об’єкт і потім прибирає запис зі стану.

## Потоки й навантаження

- WPF UI не читає й не завантажує файли синхронно.
- Scheduler спить до потрібної дати, але прокидається щонайбільше раз на 30 хвилин для актуалізації стану.
- Повне сканування виконується під час ручного запуску або коли настав графік 1–7 днів.
- Один `SemaphoreSlim` у `SyncEngine` блокує паралельні резервні копії.
- Один Windows Mutex блокує запуск другої копії застосунку.

## Обробка помилок

Тимчасові помилки піднімаються до `SyncEngine`, перетворюються на зрозумілий текст і потрапляють у журнал без токенів. Scheduler повторює невдалу фонову спробу не раніше ніж через 15 хвилин. Успішні файли вже мають SQLite-стан і не завантажуються повторно.

## Межі безпеки

- OAuth Client ID у Desktop app є публічним і може бути в інсталяції; пароль Google та приватні ключі не потрібні.
- Refresh token зберігається лише в `auth.dat`, зашифрованому DPAPI.
- Область `drive.file` обмежує застосунок файлами, які він створив або які користувач явно відкрив через нього.
- Журнал навмисно не пише HTTP body авторизації, access token або refresh token.
- Автооновлення відмовляється запускати Installer без 64-символьного SHA-256.
